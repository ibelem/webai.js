## Vision

### 10x Check

The current design generates code from CLI flags. The 10x version generates code from **intent**.

```
CURRENT (9/10):
  webai generate --task object-detection --model ./yolov8n.onnx --engine ort --framework nextjs --input camera

10x VERSION:
  webai ./yolov8n.onnx
  webai https://huggingface.co/user/repo/resolve/main/yolov8n.onnx
  webai user/repo
  -> Auto-detects: object-detection task (from output shape [1,84,8400])
  -> Auto-selects: best engine + backend for current device
  -> Auto-picks: camera input (default for detection tasks)
  -> Asks: "Which framework?" or uses last preference
  -> Generates: working Next.js app with live camera bounding boxes in 3 seconds
```

The model file IS the intent. The tool figures out everything else. Flags become overrides, not requirements.

Note: "3 seconds" refers to code generation and file output time only, excluding `npm install`, model download, and first browser render. The generation itself is pure file I/O with no network calls.

### Platonic Ideal

A developer drags a model file onto a web page. Within seconds, they're looking at a live camera feed with inference results overlaid. They didn't configure anything. The tool read the model's input/output tensors, matched them to a known task profile, selected the fastest available backend, and rendered a polished, accessible UI.

They click "Export as Next.js" and get a complete project. The generated code is so clean that a junior developer could read it and understand exactly how browser-based AI inference works. Each function has a docstring explaining not just what it does, but why.

The platonic ideal is a tool that makes web AI *obvious*. Not just possible. Obvious.

## Key Temporal Decisions

1. **Template engine:** String template literals with tagged helpers for indentation/conditionals. Example: `` code`\n  ${ifCamera(`const stream = await navigator.mediaDevices.getUserMedia(...)`)}\n` ``. Start simple, graduate to AST only if conditional nesting exceeds 3 levels.
2. **Compatibility matrix:** Approved (task x input mode). CLI rejects invalid combinations.
3. **Preprocessing implementation:** Typed-array math (works in Node + Workers), canvas-based alternative for DOM contexts.
4. **Camera inference loop:** requestAnimationFrame with frame skipping.
5. **AudioWorklet:** Separate .js file for worklet processor (Blob URL doesn't work with addModule in most browsers). Fallback to ScriptProcessorNode for older browsers.
6. **Model caching:** OPFS (Origin Private File System) instead of Cache API + Service Worker. No per-framework SW complexity. Handles large models without quota issues.
7. **Compare command:** Separate `webai compare` command, not a flag on generate.

### Task x Engine Matrix

```
                  ORT Web    LiteRT.js   WebNN API
image-classif.      Y           Y           Y
object-detection    Y           Y           Y (if ops supported)
image-segment.      Y           Y           Y (if ops supported)
feature-extraction     Y           Y           Y
ASR                 Y           Y*          -  (complex graph)
audio-classif.      Y           Y*          -  (complex graph)
TTS                 Y           -           -  (complex graph)
text-classif.       Y           Y*          Y (if ops supported)
text-generation     Y           Y*          Y (with dual session)
```

Y* = LiteRT.js support depends on model format availability (TFLite). Not all HuggingFace
models ship TFLite weights. ORT Web (ONNX) has the widest model coverage. WebNN API
requires all model ops to be WebNN-supported (check via model2webnn op compatibility).

**WebNN device type notes:** WebNN columns in this matrix apply to all three device types
(CPU, GPU, NPU). Op support may vary by device: NPU accelerators typically support a
narrower op set than GPU. The auto-selection chain (NPU → GPU → CPU → WebGPU → WASM)
handles this transparently via try/catch per device type.

**LiteRT.js maturity note:** LiteRT.js (`@litertjs/core`) provides a low-level inference
API: `loadLiteRt()` → `loadAndCompile(model, { accelerator })` → `model.run(tensor)`.
Supports WASM, WebGPU, and WebNN accelerators with device preference options
(`webNNOptions: { devicePreference: 'npu' | 'gpu' | 'cpu' }`). Phase 1 targets
LiteRT.js for CV tasks only. Audio/NLP via LiteRT.js deferred to Phase 2+ pending
model format availability (not all HuggingFace models ship TFLite weights).

### Framework x Service Worker Notes

Offline-first (--offline) service worker behavior per framework:
- `html`: Standard SW registration in generated HTML. Simplest path.
- `vanilla-vite`: SW registered in `main.js`. Vite build handles SW as static asset.
- `react-vite`: SW registered in `main.tsx`. Same as vanilla-vite.
- `nextjs`: Uses `next-pwa` or manual SW in `public/`. Requires additional `next.config.js` config. Higher complexity.
- `sveltekit`: Uses SvelteKit `$service-worker` module. Framework-native but different pattern.

Revised: --offline uses OPFS (Origin Private File System) for model caching instead of
service workers. OPFS works identically across all frameworks (no per-framework SW
integration needed). Handles large model files without Cache API quota limits. Generated
code fetches model once, stores in OPFS, loads from OPFS on subsequent visits.

## Auto-Detection Algorithm

### Task Detection from Model Shape

Shape-based detection is heuristic. Priority order:

1. **Model metadata** (ONNX model properties, custom metadata fields) -- highest confidence
2. **Output tensor shape patterns:**
   - `[1, N]` where N=1000 or N<5000 -> image-classification (confidence: high)
   - `[1, N]` where N<10 -> text-classification or small vocab (confidence: medium, prompt user)
   - `[1, C, H, W]` where output=input shape -> image-segmentation (confidence: high)
   - `[1, K, D]` where K>100 and D>4 -> object-detection/YOLO-style (confidence: high)
   - `[1, D]` where D>256 -> feature-extractionion/embeddings (confidence: medium)
3. **Input tensor shape context:**
   - 4D input [1,3,H,W] -> image task (narrows candidates)
   - 2D input [1,N] -> text/sequence task (narrows candidates)
   - 3D input [1,T,F] -> audio task (narrows candidates)

**Fallback when ambiguous (confidence < high):**
- CLI: prompt user with shortlist: "Detected possible tasks: [image-classification, feature-extractionion]. Which one?"
- If `--task` flag provided, skip detection entirely (explicit override)
- If non-interactive (piped input), fail with: "Could not auto-detect task. Use --task to specify."

### Backend Auto-Selection

Default engine: ORT Web. Default backend: auto-negotiate the best available.

**Terminology:**
- **Engine** (`--engine`): the JS inference library. ORT Web, LiteRT.js, or WebNN API.
  Determines the generated code's API calls.
- **Backend** (`--backend`): the hardware execution path. WASM (CPU only), WebGPU (GPU only),
  or WebNN (CPU, GPU, or NPU). Determines what hardware runs the math.

**Backend capabilities:**
```
Backend        Hardware    Notes
─────────────────────────────────────────────────────
wasm           CPU only    Always available. Slowest.
webgpu         GPU only    Widely available on modern browsers.
webnn-cpu      CPU         Via OS ML framework (OpenVINO, CoreML, etc.)
webnn-gpu      GPU         Via DirectML (Windows), CoreML (macOS), etc.
webnn-npu      NPU         Dedicated AI accelerator. Not available on all hardware.
```

ORT Web and LiteRT.js can use ALL backends. WebNN API engine uses only webnn-* backends.

**Generated code (ORT Web, default auto-selection):**

```javascript
// Backend auto-selection (generated code)
const providers = [];
if ('ml' in navigator) {
  // WebNN: try NPU first (dedicated AI accelerator, lowest power, highest throughput)
  // Then GPU (fast, widely available)
  providers.push({ name: 'webnn', deviceType: 'npu' });
  providers.push({ name: 'webnn', deviceType: 'gpu' });
}
if (navigator.gpu) providers.push('webgpu');             // WebGPU: GPU only
providers.push('wasm');                                   // WASM: CPU only, always available
const session = await ort.InferenceSession.create(modelPath, { executionProviders: providers });
```

**Pure WebNN API engine (`--engine webnn`):**
```javascript
// Try NPU first, fall back to GPU, then CPU
let context;
for (const deviceType of ['npu', 'gpu', 'cpu']) {
  try {
    context = await navigator.ml.createContext({ deviceType });
    console.log(`WebNN: using ${deviceType}`);
    break;
  } catch (e) {
    console.warn(`WebNN ${deviceType} not available`);
  }
}
if (!context) throw new Error('WebNN not available on this device');
```

**LiteRT.js engine (`--engine litert`):**
```javascript
import { loadLiteRt, loadAndCompile, Tensor } from '@litertjs/core';

// Initialize LiteRT WASM runtime
await loadLiteRt('./wasm/');

// Backend auto-selection: try WebNN NPU → WebNN GPU → WebGPU → WASM
const accelerators = [];
if ('ml' in navigator) {
  accelerators.push({ accelerator: 'webnn', webNNOptions: { devicePreference: 'npu' } });
  accelerators.push({ accelerator: 'webnn', webNNOptions: { devicePreference: 'gpu' } });
}
if (navigator.gpu) accelerators.push({ accelerator: 'webgpu' });
accelerators.push({ accelerator: 'wasm' });

let model;
for (const opts of accelerators) {
  try {
    model = await loadAndCompile(modelPath, opts);
    console.log(`LiteRT: using ${opts.accelerator}`);
    break;
  } catch (e) {
    console.warn(`LiteRT ${opts.accelerator} not available`);
  }
}

// Run inference
const input = new Tensor(preprocessedData, [1, 3, 224, 224]);
const outputs = await model.run(input);
const result = await outputs[0].data();
```

**CLI flags (all flags have short aliases):**

```
Flag               Short   Values                         Default
────────────────────────────────────────────────────────────────────
--engine           -e      ort, litert, webnn             ort
--backend          -b      wasm, webgpu, webnn-cpu,       (auto)
                           webnn-gpu, webnn-npu
--task             -t      image-classification, etc.     (auto-detect)
--model            -m      path, URL, or HuggingFace ID   (required)
--framework        -f      html, vanilla-vite, nextjs,    html

--input            -i      file, camera, video, mic,      (task default)
                           screen
--mode                     raw, compact                   raw
--lang             -l      js, ts                         js
--output           -o      output directory path          ./output/
--offline                  (flag, no value)               off
--theme                    dark, light                    dark
--verbose          -v      (flag, no value)               off
--force                    (flag, no value)               off
```

Engine (which JS inference library, `-e`):
- `-e ort` — ORT Web (default). Generated code uses `ort.InferenceSession`.
- `-e litert` — LiteRT.js. Generated code uses `@litertjs/core`: `loadLiteRt()` → `loadAndCompile()` → `model.run()`.
- `-e webnn` — Pure WebNN API. Generated code uses `navigator.ml.createContext()`
  + `MLGraphBuilder`. No ORT Web or LiteRT.js dependency.

Backend (which hardware execution path, `-b`, optional override):
- `-b wasm` — Force WASM (CPU only).
- `-b webgpu` — Force WebGPU (GPU only).
- `-b webnn-npu` — Force WebNN NPU.
- `-b webnn-gpu` — Force WebNN GPU.
- `-b webnn-cpu` — Force WebNN CPU.
- No `-b` flag → auto-negotiate best available.

Shorthand for `-e webnn`: since the engine is already WebNN, `-b npu`,
`-b gpu`, `-b cpu` are accepted (no `webnn-` prefix needed).

**Model input formats:**
- `-m ./model.onnx` — local file path
- `-m https://huggingface.co/user/repo/resolve/main/model.onnx` — direct HuggingFace URL
- `-m user/repo` — HuggingFace model ID (resolves to best available ONNX/TFLite file)
- Positional: `webai ./model.onnx` or `webai https://hf.co/.../model.onnx` or `webai user/repo`

When the model source is a URL or HF model ID:
- CLI fetches the model to a temp location for metadata parsing and task auto-detection
- Generated code references the URL directly (not a local path)
- When combined with `--offline`, generated code uses OPFS caching: fetch once from URL, load from cache after
- HuggingFace URL normalization: `/blob/` rewritten to `/resolve/` for direct download (same pattern as model2webnn)
- HuggingFace mirror fallback: auto-fallback to `hf-mirror.com` if primary URL times out (5s HEAD pre-check)

**Examples with short flags:**
```bash
webai ./yolov8n.onnx -f nextjs                    # local file, auto-detect everything
webai https://huggingface.co/user/repo/resolve/main/yolov8n.onnx -f nextjs  # HF URL
webai user/repo -f nextjs                          # HF model ID, auto-resolve file
webai generate -t object-detection -m ./yolov8n.onnx -e ort -f nextjs -i camera
webai generate -m ./model.onnx -e ort -b webgpu   # force WebGPU backend
webai generate -m ./model.onnx -e webnn -b npu    # pure WebNN API, force NPU
webai compare ./model.onnx                         # benchmark all backends
```

## Testing Strategy

### Generator Tests (testing webai.js itself)

1. **Unit tests** for preprocessing functions: image-resize, normalize, to-nchw, softmax, topk, argmax, nms. Pure functions on typed arrays, easy to test with known inputs/outputs.
2. **Snapshot tests** for code emitters: each (task, engine, framework, input, mode) combination produces deterministic output. Snapshot the generated file contents. Detect unintended changes.
3. **Integration tests** for the assembler: given a task profile + config, verify the assembled output is syntactically valid JS/TS (parse with TypeScript compiler API, no need to execute).
4. **CLI tests**: invoke `webai generate` with various flag combinations, verify correct files are created in the output directory with expected structure.
5. **Template build tests**: for each framework template, verify the generated project can `npm install && npm run build` without errors. Run in CI with a test model.

### Generated Code Correctness

6. **Golden output tests**: for image-classification + ORT Web + html, maintain a reference model (e.g., MobileNetV2 quantized, ~5MB). Run the generated HTML in Playwright, feed a test image, verify top-1 classification matches expected label.
7. **Cross-backend parity**: same model + same input should produce identical results across ORT Web backends (WASM vs WebGPU). Tolerance: <1e-5 for float32 outputs.

### Test Framework

Vitest (matches the Vite-based build, supports TypeScript natively, fast).

### Compare Command Spec

`webai compare ./model.onnx` generates a single HTML benchmark page:
- **Metrics measured:** first-inference latency (cold), steady-state latency (warm, avg of 10 runs), throughput (inferences/sec), peak JS heap
- **Output:** HTML page with comparison table + bar charts. Also `--json` flag for machine-readable output.
- **Backends tested (Phase 2):** WASM, WebGPU, WebNN (CPU), WebNN (GPU), WebNN (NPU).
  Each WebNN device type is benchmarked separately because performance varies dramatically
  (NPU optimized for sustained throughput, GPU for parallel ops, CPU as baseline).
- **Backend unavailable:** Skip with warning per backend ("WebNN (NPU): not available in this browser"). Test remaining backends.
- **Scope:** ORT Web backends only in Phase 2. LiteRT.js and pure WebNN API comparison in Phase 3.

## Error Handling for Auto-Detection

| Scenario | Behavior |
|----------|----------|
| Model file not found / unreadable | Fail fast: "Model not found: {path}" |
| Model file corrupted / invalid format | Fail fast: "Could not parse model: {path}. Expected ONNX (.onnx) or TFLite (.tflite)" |
| Task auto-detect: high confidence | Use detected task, print: "Detected task: image-classification (from output shape [1,1000])" |
| Task auto-detect: medium confidence | Interactive: prompt user with shortlist. Non-interactive: fail with suggestion. |
| Task auto-detect: no match | Fail: "Could not detect task from model shape. Use --task to specify." |
| Backend not available in browser | Generated code falls through EP chain (WebNN NPU -> WebNN GPU -> WebGPU -> WASM). Console warning for each unavailable backend. |
| WebNN device type unavailable | Fallback within WebNN: NPU -> GPU -> CPU. If all WebNN devices fail, fall through to WebGPU -> WASM. Status bar shows actual device used. |
| Model has unsupported WebNN ops | CLI warning at generation time (if detectable from op list). Generated code includes try/catch around WebNN session creation with ORT Web WASM fallback. |
| HF config fetch fails (network) | CLI warning: "Could not fetch HF config, using task defaults." Continue generation. |
| HF URL model fetch fails (network) | Fail fast: "Could not fetch model from {url} — check URL and network connection." Suggest using a local file instead. |
| HF URL returns non-model content (HTML) | Fail fast: "URL returned HTML, not a model file. For HuggingFace, use the /resolve/ URL (not /blob/)." Auto-rewrite attempted first. |
| HF model ID has no ONNX/TFLite files | Fail fast: "No .onnx files found in {model-id}. This model may not have ONNX weights. Try downloading and converting manually." |
| HF mirror fallback triggered | CLI info: "Primary HuggingFace URL slow, trying mirror..." Transparent to user, falls back automatically. |
| Invalid flag combination | CLI error: "mic input is not supported for image-classification. Try --input camera" (from compatibility matrix) |

Key contract: Layer 1 produces `CodeBlock[]` (engine-specific code chunks).
Layer 2 consumes `CodeBlock[]` and produces `GeneratedFile[]` (complete files
in framework scaffold). Layer 3 writes files. Layers never call each other
directly. The assembler orchestrates: `config → L1(config) → L2(config, blocks) → L3(files)`.
9. **ONNX metadata parsing:** `protobufjs` with official `onnx.proto3` schema. Header-only parsing (skip weight data). TFLite: `flatbuffers` with TFLite schema.
10. **Tensor input/output names:** Parsed from ONNX/TFLite metadata alongside shapes. Same parse step as task auto-detection.
11. **Generated code error handling:** Exactly 3 inline error handlers: camera/mic permission denied, model 404, backend unavailable. Everything else is user's concern.
12. **Input validation:** HF URL domain validation (huggingface.co only) + model path validation (resolve, verify extension). CLI boundary security.
13. **CLI --verbose flag:** Prints resolver trace (each step + resolved values), template selection, files written. Essential for debugging wrong preprocessing.
14. **Model caching via OPFS:** Origin Private File System replaces Cache API + Service Worker for --offline flag. No per-framework SW complexity. Handles large models.
15. **Web UI in Phase 2:** Simple dropdown-based (not visual node builder). Code preview with Monaco. "Try it" button with sandboxed iframe.
16. **Auto-calibrated frame skipping:** Camera inference loop measures inference time and auto-adjusts frame skip rate to match model throughput.
17. **Header-only model parsing:** Delegates to model2webnn's `parseOnnxMetadata()` / `parseTfliteMetadata()` (added in commit 93975dc, perf-fixed in commits 1a0a6dc + 5dff577). ONNX: zero-copy protobuf scanner reads only graph I/O fields (5, 11, 12), skips weight data via length-varint jump in O(1). TFLite: FlatBuffers reader never touches buffers vector (field 4). 2GB model parses in milliseconds. No duplicate protobufjs/flatbuffers in webai.js.
18. **Framework template currency:** Each template uses current framework best practices (Next.js app router, Svelte 5 runes, React 19). Maintenance commitment.
19. **Responsive design:** All generated apps responsive by default. CSS viewport-width video/canvas. Touch-friendly overlays.
20. **Generated code versioning:** Snapshot pattern. --force required to overwrite. Documented in generated README.
21. **Cross-repo compatibility:** Semver strictly. model2webnn pins @webai/core to major version range.
22. **AudioWorklet:** Separate .js file for worklet processor (Blob URL doesn't work with addModule). ScriptProcessorNode fallback.
23. **WebNN device types (from design review):** WebNN has three device types: CPU, GPU, NPU. All plans, UI, status bars, compare benchmarks, CLI output, and generated code must distinguish between them. The auto-selection chain is: WebNN NPU → WebNN GPU → WebGPU → WASM. CLI override: `--backend webnn-npu`, `--backend webnn-gpu`, `--backend webnn-cpu`. Pure WebNN API code negotiates `createContext({ deviceType })` with per-device try/catch. ORT Web EP uses `{ name: 'webnn', deviceType: 'npu' | 'gpu' | 'cpu' }`.
24. **Engine + Backend terminology (from design review):** `--engine` selects the JS inference library (ort, litert, webnn). `--backend` selects the hardware execution path (wasm, webgpu, webnn-cpu, webnn-gpu, webnn-npu). WASM is CPU only, WebGPU is GPU only, WebNN covers CPU/GPU/NPU. Replaces the overloaded `--runtime` flag which conflated two distinct concepts.
25. **CLI framework (from eng review):** commander. Declarative subcommands (`generate`, `compare`, `tasks`, `models`, `frameworks`), auto-generated help, built-in support for short aliases (-e, -b, -t). Layer 1, same choice as Transformers.js.
26. **Build tool (from eng review):** tsup (esbuild-based). One `tsup.config.ts` per package. ESM+CJS dual output for `@webai/core`. ESM-only for CLI. Zero-config TypeScript compilation.
31. **CLI performance budget (from eng review):** `webai generate` completes in under 3 seconds for `--engine ort` and `--engine litert` with any model file up to 2GB. Breakdown: model metadata parsing < 500ms (zero-copy protobuf scanner via model2webnn, see Decision #32), config resolution < 50ms, code emission < 100ms, file write < 50ms. Remaining budget covers Node.js startup + commander parsing. CI regression test: measure and assert total CLI time stays under budget. **Exception:** `--engine webnn` is exempt from the 3s budget (see Decision #34) because model2webnn's `convert()` does full model processing.
30. **LiteRT.js API (from eng review):** LiteRT.js (`@litertjs/core`) is the low-level inference engine, NOT the MediaPipe tasks API. API: `loadLiteRt(wasmPath)` → `loadAndCompile(modelPath, { accelerator })` → `model.run(tensor)`. Supports `accelerator: 'wasm' | 'webgpu' | 'webnn'` with `webNNOptions: { devicePreference: 'npu' | 'gpu' | 'cpu' }`. MediaPipe is a high-level framework (same layer as Transformers.js) and is not used by webai.js.
29. **Resolver fallback warning (from eng review):** When the preprocessing resolver falls back to task defaults (no HF config, no known model profile), the CLI prints: "⚠ Using task defaults for preprocessing — verify mean/std match your model." AND the generated code includes a comment: `// WARNING: Preprocessing uses ImageNet defaults (mean=[0.485,0.456,0.406], std=[0.229,0.224,0.225]). If results look wrong, check these values against your model's training config.` Both warnings only appear when task defaults are actually used, not when HF config or model profile provided exact values.
28. **Preprocessing dual-use (from eng review):** `@webai/core` has real, executable, Vitest-tested preprocessing functions (resizeImage, softmax, topK, etc.). The CLI's preprocess-emitter mirrors the same logic as template literals, emitting standalone JS/TS. Cross-verification: snapshot tests feed identical inputs to both the real function and eval(emitted code), assert outputs match. Two copies of the logic, one test that binds them.
27. **Package boundary (from eng review):** `@webai/core` = knowledge (preprocessing function implementations, postprocessing functions, task profiles, config resolver, model parser). `packages/cli` = ALL codegen (preprocess emitter, postprocess emitter, inference emitters, assembler, framework IO emitters, file writer). model2webnn imports `@webai/core` for the function catalog and task profiles, not for codegen. Clean separation: core knows WHAT to do, CLI knows HOW to emit it as code.
32. **ONNX metadata parsing perf (from eng review outside voice, RESOLVED):** Originally `parseOnnxMetadata()` called `ModelProto.decode(buffer)` which deserialized the entire protobuf. Fixed in model2webnn commits `1a0a6dc` + `5dff577`: replaced with a hand-rolled zero-copy protobuf scanner. The new implementation uses `pbVarUint`/`pbSkip`/`pbLenView` helpers to scan only GraphProto fields 5 (initializer names), 11 (inputs), 12 (outputs). Weight data (`raw_data` field 9) is skipped in O(1) via a length-varint read. A 2GB model with 200 initializers reads only a few KB of actual bytes, no heap allocations for weight data. The <3s budget now holds for all model sizes. Pre-opset-9 models handled correctly (initializer names filtered from `graph.input`).
33. **CodeBlock auxiliary files (from eng review outside voice):** WebNN engine (`--engine webnn`) via model2webnn's `convert()` returns a `buildGraph()` function (potentially 50K+ lines for large models) plus a WGWT binary weights file plus a manifest JSON. This is fundamentally different from ORT Web / LiteRT which just reference a model file. The CodeBlock interface needs an `auxiliaryFiles` field: `CodeBlock = { id, code, imports, exports, auxiliaryFiles?: GeneratedFile[] }`. The inference-webnn.ts emitter populates `auxiliaryFiles` with the WGWT binary and manifest. Layer 2 (IO emitter) passes them through to Layer 3 (writer) without needing engine-specific knowledge. This preserves the "layers don't call each other" contract.
34. **WebNN generation exempt from 3s budget (from eng review outside voice):** model2webnn's `convert()` does full protobuf/flatbuffers parse, free dimension resolution, shape propagation, constant folding (up to 10 iterations), MatMulNBits fixup, weight packing, and code generation. For large models this can take 10+ seconds. The 3-second CLI performance budget (Decision #31) applies to `--engine ort` and `--engine litert` only. `--engine webnn` is exempt. CLI output for WebNN shows a progress indicator during conversion. Document this in CLI help text.
35. **Cross-verification test isolation (from eng review outside voice):** Tests T20-T23 that cross-verify emitted code against real functions use `vm.runInNewContext()` with a controlled global, not raw `eval()`. This prevents scope leaks and provides a clean execution environment. Only the JS emit path is tested via execution; the TS emit path is verified via TypeScript compiler API parse (T29-T30). Emitted preprocessing functions use typed-array-only implementations (no DOM APIs like ImageData or OffscreenCanvas) so they run in both Node and browser contexts.
36. **Test model strategy (from eng review outside voice):** No 5MB model committed directly to git. Instead: (a) Generate a minimal synthetic ONNX file (~1KB, valid header, correct shapes, random weights) for unit tests T1-T9. Committed to `tests/fixtures/`. (b) The golden output test model (MobileNetV2 quantized, ~5MB) is downloaded in CI via a setup step and cached. Never in the repo. (c) If any model fixture exceeds 100KB, use git-lfs. This prevents git history bloat as more model formats are needed in Phase 2+.
37. **No circular dependency (from eng review outside voice):** The plan previously stated "model2webnn depends on @webai/core for preprocessing function catalog (future)." This creates a circular npm dependency (A depends on B depends on A). Revised: model2webnn NEVER depends on @webai/core. If model2webnn needs preprocessing functions in the future, it either (a) vendors/reimplements what it needs, or (b) takes preprocessing functions as injected callbacks. Dependency direction is strictly one-way: @webai/core → model2webnn, packages/cli → model2webnn + @webai/core.
38. **Snapshot selection criteria (from eng review outside voice):** Not all 2700 (5 frameworks x 3 engines x 9 tasks x 5 inputs x 2 modes x 2 langs) combinations get snapshots. Selection strategy: one "golden" combo per engine (3), one per framework (5), one per input mode (5), one per mode/lang variant (4). That is ~17 representative snapshots. New snapshots are added when: (a) a new engine/framework/input is added, or (b) a bug is found in a specific combination. T24-T28 cover the Phase 1a subset. Selection criteria documented in `tests/README.md`.
39. **Dev infrastructure in Phase 1a (from eng review outside voice):** Phase 1a includes: (a) GitHub Actions CI config (lint + typecheck + test on push/PR), (b) ESLint + Prettier config (shared across packages, also formats emitted code templates), (c) npm workspace protocol (`"@webai/core": "workspace:*"` in packages/cli/package.json), (d) `packages/cli/package.json` bin entry pointing to compiled CLI, (e) Node.js >=20 in engines field + `.nvmrc`, (f) TypeScript strict mode in both packages.
40. **Online model support (CLI + Web UI):** Both CLI and Web UI accept three model input formats: (a) local file path (`./model.onnx`), (b) direct HuggingFace URL (`https://huggingface.co/user/repo/resolve/main/model.onnx`), (c) HuggingFace model ID (`user/repo`, auto-resolves to best available ONNX or TFLite file). The resolver classifies the input as `local-path | url | hf-model-id` and normalizes to an `ArrayBuffer` for metadata parsing. For URLs and model IDs, the CLI fetches the model to a temp location; generated code references the URL directly. HuggingFace URL normalization rewrites `/blob/` to `/resolve/` for direct download (same pattern as model2webnn's `transformHuggingFaceUrl()`). Mirror fallback: auto-fallback to `hf-mirror.com` if primary HuggingFace URL times out (5s HEAD pre-check, same pattern as model2webnn). When combined with `--offline`, generated code uses OPFS caching: fetches from URL on first load, serves from OPFS cache on subsequent loads. Web UI: URL paste field with same normalization + streaming progress bar. HF model ID resolution: fetch `https://huggingface.co/api/models/{id}` to list files, pick the first `.onnx` file (or `.tflite` if `--engine litert`). Resolves Open Question #1.

### model2webnn Integration Contract (from eng review)

webai.js uses model2webnn in two ways:

**1. Model metadata parsing (used by @webai/core for task auto-detection):**
```typescript
import { parseOnnxMetadata, parseTfliteMetadata, detectFormat } from 'model2webnn';
import type { ModelMetadata } from 'model2webnn';

// Lightweight metadata-only parse — sync, skips weight data entirely.
// A 2GB ONNX model parses in milliseconds (no typed-array allocations for weights).
const buffer = new Uint8Array(fs.readFileSync(modelPath));
const format = detectFormat(buffer);
const metadata: ModelMetadata = format === 'onnx'
  ? parseOnnxMetadata(buffer)
  : parseTfliteMetadata(buffer);

// metadata.inputs[0].shape → [1, 3, 224, 224] → image task
// metadata.outputs[0].shape → [1, 1000] → classification
// metadata.inputs[0].dataType → 'float32'
// No duplicate protobufjs/flatbuffers parsing. Zero new dependencies.
```

**2. WebNN code generation (inference-webnn.ts emitter):**
```typescript
import { convert } from 'model2webnn';
import type { ConvertResult, OperatorCoverage } from 'model2webnn';

// inference-webnn.ts produces CodeBlock[] for --engine webnn
const result: ConvertResult = await convert(buffer, {
  format: config.lang === 'ts' ? 'typescript' : 'javascript',
  modelName: config.modelName,
  freeDimensionOverrides: config.freeDimOverrides,
});

// result.code → buildGraph(context, weights) function (pure MLGraphBuilder calls)
// result.weights → WGWT binary file (written alongside generated code)
// result.manifest → tensor metadata JSON (for weight loading)
// result.coverage → op support analysis (warn if unsupportedOps > 0)

// inference-webnn.ts wraps result.code with:
//   - Device type negotiation (NPU → GPU → CPU)
//   - MLContext creation
//   - Weight loading from WGWT file
//   - MLTensor pre-allocation (optimization pattern #1)
//   - dispatch() + readTensor() calls
// These become the CodeBlock with id='inference'
```

**Dependency direction (strictly one-way, no circular deps):**
`@webai/core` depends on model2webnn for parsing.
`packages/cli` depends on model2webnn for WebNN codegen + on `@webai/core` for everything else.
model2webnn NEVER depends on `@webai/core` (see Decision #37). If model2webnn
needs preprocessing in the future, it vendors or takes callbacks.

**Op coverage gate:** If `result.coverage.unsupportedOps > 0`, CLI prints warning:
"Warning: {N} unsupported WebNN ops ({opTypes}). Model may not run correctly.
Consider --engine ort for full op support."

## Visual Language for Generated Apps

### Design Philosophy

Generated apps should feel like developer tools, not consumer apps. Think Chrome DevTools
meets a camera feed. Technical but clean. Every element earns its pixels.

**Theme:** Dark by default (`--theme light` for light mode). Dark recedes behind the
camera feed / image / waveform. Light mode inverts the palette but keeps the same layout.

**Typography:** System monospace stack for metrics, labels, and status (`ui-monospace,
'Cascadia Code', 'Source Code Pro', monospace`). System sans-serif for UI text
(`system-ui, -apple-system, sans-serif`). No custom font loading in generated apps.

**Color system (dark theme):**
```
--bg:           #0a0a0a     (near-black background)
--surface:      #1a1a1a     (panels, controls)
--text:         #e5e5e5     (primary text)
--text-muted:   #737373     (secondary text, labels)
--accent:       #3b82f6     (blue, interactive elements)
--success:      #22c55e     (green, loaded/ready states)
--warning:      #eab308     (yellow, fallback/degraded)
--error:        #ef4444     (red, failures)
```

**Spacing:** 4px base unit. 8px, 12px, 16px, 24px, 32px scale. Tight for developer tools.

### Result Element Visual Specs

#### Bounding Boxes (Object Detection)

- **Border:** 2px solid, color by class. First 10 classes use a fixed palette:
  `['#3b82f6', '#22c55e', '#eab308', '#ef4444', '#a855f7', '#06b6d4', '#f97316', '#ec4899', '#14b8a6', '#8b5cf6']`
- **Fill:** Same color at 10% opacity (semi-transparent fill helps see overlapping boxes)
- **Label:** Class name + confidence %. White text on class-color background pill.
  Positioned above the box top-left. Font: 12px monospace. Padding: 2px 6px.
- **Low confidence (<50%):** Dashed border instead of solid. Label text at 60% opacity.
- **No shadow, no rounded corners on boxes.** Boxes are data, not decoration.

#### Confidence Bars (Image Classification)

- **Layout:** Horizontal bars, left-aligned. Label left, bar right, percentage right-aligned.
- **Bar height:** 24px. Gap between bars: 4px. Compact, scannable.
- **Bar color:** Gradient from accent (#3b82f6) at 100% to muted gray at 0%.
  Top result uses accent color. Others use progressively lighter shades.
- **Top result:** Bold text, full opacity. Others: normal weight, 70% opacity.
- **Show top-5 by default.** Results below 1% hidden.
- **Format:** `golden retriever  ████████████████  92.3%`
  Monospace percentage, right-aligned for easy scanning.

#### Segmentation Masks

- **Overlay:** Semi-transparent colored mask over the image/camera feed.
  Each class gets a color from the same 10-color palette as bounding boxes.
- **Opacity:** 40% fill. Enough to see the mask, enough to see the image beneath.
- **Border:** 1px solid at 60% opacity between adjacent classes. Helps distinguish regions.
- **Hover (file input only):** Hovering a region shows class name + pixel count in tooltip.

#### Audio Waveform (ASR, Audio Classification)

- **Style:** Amplitude line graph, not frequency bars. Thin 1px line, accent color.
- **Background:** Subtle grid lines at 50% and 100% amplitude. Grid color: #1a1a1a.
- **Height:** 80px. Enough to see shape, not dominant. The transcription text is the hero.
- **Animation:** Scrolls left as new audio arrives. 60fps render via canvas.
- **Recording indicator:** Small red dot (pulsing) next to the waveform when mic is active.

#### Status Bar

- **Position:** Bottom of viewport, fixed. Full width. Height: 32px.
- **Content:** Model name · inference latency · backend. Separated by · (middle dot).
  Examples: `yolov8n · 8ms · WebNN (NPU)` or `mobilenet-v2 · 23ms · WebGPU` or `whisper-tiny · 120ms · WASM`
- **Style:** Monospace, 12px, text-muted color. Background: surface color with 90% opacity
  (subtle blur-behind if backdrop-filter supported).
- **Backend indicator:** Small colored dot before backend name.
  Green = best available (NPU or GPU), yellow = fallback (WASM), red = error.
- **WebNN device types:** Always show the specific device type in parentheses:
  "WebNN (NPU)", "WebNN (GPU)", "WebNN (CPU)". Never just "WebNN" alone.

## Responsive & Accessibility Specifications

These specs become generated code. Every template includes them.

### Viewport Breakpoints

```
BREAKPOINT     WIDTH         LAYOUT CHANGES
─────────────────────────────────────────────────────────────────
Mobile         < 640px       - File input: image + results stack vertically
                             - Camera: feed fills full viewport (no side panels)
                             - Controls: bottom-anchored horizontal strip
                             - Status bar: collapses to icon + latency only
                             - Confidence bars: full-width below image

Tablet         640-1024px    - File input: side-by-side (image left, results right)
                             - Camera: feed 85% viewport, controls bottom-right
                             - Touch targets: 44px minimum (enforced)

Desktop        > 1024px      - Full layout as specified in hierarchy diagrams
                             - Hover states enabled on bounding boxes / results
                             - Keyboard shortcuts shown in tooltips
```

No "hamburger menu." Generated apps are single-screen. There is nothing to navigate.
Mobile just reflows the layout.

### Keyboard Navigation

Generated apps include keyboard support in this tab order:

**Camera / realtime apps:**
1. Tab to controls: pause/resume (Space), toggle labels (L key)
2. Escape: stop inference, release camera/mic
3. No focus trap. Tab out exits the app controls, reaches the browser chrome.

**File input apps:**
1. Tab to drop zone. Enter/Space opens file picker.
2. After results: Tab moves through result items (confidence bars are focusable).
3. Tab to "Choose another file" button.

**All generated apps:**
- `prefers-reduced-motion: reduce` — disable waveform animation, disable overlay
  transitions, keep inference running (it's functional, not decorative).
- Skip link: "Skip to results" (keyboard-only visible) for screen reader users.

### Screen Reader Accessibility

**Inference results as live region:**
```html
<div role="status" aria-live="polite" aria-atomic="true" id="results">
  <!-- Updated by inference loop -->
</div>
```

- `aria-live="polite"` — announces results when screen reader is idle, not mid-sentence.
- `aria-atomic="true"` — reads the full result, not incremental changes.
- For camera detection: announces "N objects detected: [top classes]" every 3 seconds
  (not every frame, that would be overwhelming).
- For classification: announces "Top result: golden retriever, 92.3% confidence."
- For ASR: `aria-live="assertive"` on the transcription, streams text naturally.

**ARIA landmarks in generated HTML:**
```html
<main>              <!-- The primary content (camera feed, image, results) -->
<aside>             <!-- Status bar, controls -->
<footer>            <!-- "Generated by webai.js" -->
```

**Camera/mic permission:**
```html
<button aria-label="Enable camera for object detection">Enable Camera</button>
```
Not just "Enable Camera" — the label includes what it's for.

**Bounding box overlay:**
Canvas overlays are not accessible (canvas content is invisible to screen readers).
The live region with text results is the accessible equivalent. This is documented
in the generated code with a comment explaining why.

## Interaction State Specifications

Every generated app template must implement these states consistently. Describes what the
user SEES, not backend behavior.

### Model Loading States

| State | User sees |
|-------|-----------|
| **Downloading** | Progress bar with model size + percentage. "Loading model.onnx (23 MB)..." Center screen, no other UI visible yet. |
| **OPFS cache hit** | "Loaded from cache (42ms)" — near-instant, brief flash of status then straight to ready. |
| **Parse/compile** | "Preparing model..." after download completes. Brief (usually <1s), but shown to explain the gap between download complete and first inference. |
| **Ready** | Controls and input UI fade in. Status bar shows model name + size. The transition from loading to ready should feel like "opening a door." |
| **Download error** | "Could not load model.onnx — file may be corrupted or server unreachable. Check the model path and try refreshing." Centered, with retry button. |

### Camera/Mic Permission States

| State | User sees |
|-------|-----------|
| **Pre-permission** | Centered message explaining WHY: "This app uses your camera to detect objects in real time. No video is recorded or sent anywhere." + "Enable Camera" button. NOT a blank screen waiting for the browser dialog. |
| **Browser dialog** | Overlay dims the background. App waits. No spinner (the browser dialog IS the interaction). |
| **Granted** | Camera feed appears immediately. Status shows resolution + FPS. Controls fade in. |
| **Denied** | "Camera access was denied. This app needs camera access to run object detection. [How to enable in Chrome/Firefox/Safari]" — with actual browser-specific instructions (generated per-browser or with a generic "check your browser settings" link). |
| **Mic equivalent** | Same pattern. "This app uses your microphone for speech recognition. Audio is processed locally and never leaves your browser." |

### Inference States

| State | User sees |
|-------|-----------|
| **First inference** | "Running first inference..." — first run is slower (model warmup). Show this to set expectation. Then transition to steady state. |
| **Steady state** | Results updating live. Latency + FPS in status bar. Bounding boxes / labels / masks / text streaming. |
| **No results** | Camera still running, but subtle "No detections" text. NOT a blank canvas. For classification: "No confident prediction." For ASR: cursor blinking, no text yet. |
| **Low confidence** | Detections below threshold shown with lower opacity / dashed border. User knows the model is working but unsure. |
| **Inference error** | "Inference failed. Backend: {backend}. Try refreshing the page." One-time error, not per-frame. |

### Backend Fallback States

| State | User sees |
|-------|-----------|
| **Best backend** | Status bar: "WebNN (NPU)" or "WebNN (GPU)" or "WebGPU" (green indicator). Always shows the specific backend, not just "WebNN." |
| **Fallback occurred** | Subtle toast explaining the chain: "WebNN NPU unavailable, using WebNN GPU" or "WebNN not available, fell back to WebGPU" or "Using WASM (slowest)." Toast auto-dismisses after 5s. Status bar shows actual backend. |
| **All backends failed** | "Could not initialize inference engine. Your browser may not support WebAssembly." (This should be extremely rare.) |

### File Input States

| State | User sees |
|-------|-----------|
| **Empty / waiting** | Drop zone: dashed border, file icon, "Drop an image here or click to browse." Visually inviting, not clinical. |
| **File selected** | Image thumbnail replaces drop zone. "Processing..." briefly while preprocessing runs. |
| **Results ready** | Image + results side-by-side (or stacked on mobile). Classification: confidence bars. Detection: image with bounding boxes drawn. |
| **Wrong file type** | "Unsupported file. Try JPG, PNG, or WebP." Drop zone stays visible for retry. |
| **New file** | Previous results clear. New image loads. Smooth transition, no page flicker. |

## Worktree Parallelization Strategy (from eng review)

| Step | Modules touched | Depends on |
|------|----------------|------------|
| A: Core preprocessing | packages/core/src/preprocess/, packages/core/src/postprocess/ | — |
| B: Model parser + task detection | packages/core/src/model-parser/, packages/core/src/tasks/ | — |
| C: Config resolver | packages/core/src/config/ | A (task profiles reference preprocess chains) |
| D: Inference emitters (Layer 1) | packages/cli/src/emitters/ | A, B (emitters reference preprocess + task shapes) |
| E: Framework IO emitters (Layer 2) | packages/cli/src/frameworks/ | D (consumes CodeBlock[]) |
| F: Assembler + CLI + Writer | packages/cli/src/assembler.ts, cli.ts, writer.ts | D, E |
| G: Dev infrastructure + CI | root config files, .github/ | — |

**Parallel lanes:**
- Lane 1: G (dev infra) — independent, can start immediately
- Lane 2: A (preprocess) + B (model parser) in parallel — no shared modules
- Lane 3: C (resolver) — depends on A completing
- Lane 4: D (emitters) — depends on A + B
- Lane 5: E (frameworks) — depends on D
- Lane 6: F (assembler + CLI) — depends on D + E

**Execution order:**
1. Launch Lane 1 (G) + Lane 2 (A + B) in parallel worktrees
2. Merge Lane 1. Merge A. Start Lane 3 (C). Merge B. Start Lane 4 (D) — D needs both A + B
3. Merge D. Start Lane 5 (E)
4. Merge E + C. Start Lane 6 (F)

**Conflict flags:** Lanes A and B both write to `packages/core/` but different subdirectories (preprocess/ vs model-parser/). Low conflict risk. Lanes D and E both write to `packages/cli/` but different subdirectories. Low conflict risk.

**Realistic recommendation:** 3 parallel worktrees max. Lane 1 (G) is quick and merges first. Then A+B in parallel. Then C→D→E→F sequential (tight dependencies).
