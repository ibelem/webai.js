## Problem Statement

Web developers who want to run AI models locally in the browser face a fragmented, bloated landscape:

- **Transformers.js** bundles ~500KB+ with 27 pipelines, follows Python class hierarchies (AutoModel, AutoTokenizer), and offers no per-task tree-shaking. You import the castle to use one brick.
- **MediaPipe Web** is highly optimized but limited to CV tasks. No support for NLP, audio, or modern Transformer architectures.
- **model2webnn** generates clean, self-contained WebNN inference code from ONNX/TFLite models, but outputs pure inference graphs with zero pre/post-processing. Developers must write their own image normalization, tokenization, NMS, softmax decoding.
- **Raw ONNX Runtime Web / LiteRT.js** require developers to manually handle tensor creation, preprocessing, and output decoding. The APIs are similar but not identical.

The result: developers either import a massive library they don't need, or copy-paste preprocessing code from examples and hope it works.

## What Makes This Cool

A CLI and Web UI that takes a HuggingFace model (or any ONNX/TFLite file), a task type, and a target runtime, and generates the **exact minimal JavaScript code** needed to run that model end-to-end in a browser. Pre-processing, inference, post-processing. Nothing more.

The generated code has zero runtime dependencies beyond the inference engine itself. It's readable, editable, and standalone. Two modes: raw (documented, instructive) and compact (minified, production-ready). The developer gets Lego bricks, not a pre-assembled castle.

## Constraints

- Must work with all three inference runtimes: ONNX Runtime Web (WASM/WebGPU/WebNN), LiteRT.js (WASM/WebGPU/WebNN), and pure WebNN API
- Pre/post-processing functions must be atomic, composable, and independently usable
- Generated code must have no dependency on webai.js itself — it's standalone JS/TS
- Must follow HuggingFace task taxonomy for model organization
- Must work WITHOUT HuggingFace config files (many models, especially MediaPipe and custom ONNX, have no preprocessor_config.json)
- Task-driven preprocessing selection with sensible defaults; HF configs enhance but don't gate
- CLI must be installable via npm (`npx webai generate ...`)
- On-device only — no cloud AI inference

## Premises

1. **Code generation > library distribution** for web AI pre/post-processing. Generating minimal, standalone JS per task eliminates the bundle-size problem entirely.
2. **Three-runtime abstraction is tractable.** ORT Web, LiteRT.js, and WebNN API have similar load/run patterns. A thin unified abstraction works.
3. **Pre/post-processing is the actual bottleneck, not inference.** The inference runtimes are mature. The glue code is where developers struggle.
4. **HuggingFace task taxonomy is the right organizing principle.** It's what the model ecosystem already uses.
5. **model2webnn's codegen architecture can be extended or consumed.** Its IR, operator registry, and WGWT weight format are proven.
6. **"Do nothing" (copy-paste from examples) is the real competitor.**
7. **Task-driven > config-driven for preprocessing selection.** Many models (MediaPipe, custom ONNX, TFLite) lack HuggingFace metadata. The tool must work from task type + sensible defaults, with HF configs as an optional enhancement.

## Approaches Considered

### Approach A: Standalone Code Generator
New project from scratch. Clean architecture, no legacy constraints. But duplicates model2webnn patterns and requires maintaining two separate codegen tools.
Effort: L | Risk: Medium | Completeness: 8/10

### Approach B: Extend model2webnn
Add pre/post-processing codegen directly into model2webnn. Fastest path, reuses all existing infrastructure. But model2webnn's identity is "model to WebNN" — adding ORT/LiteRT changes what the project IS.
Effort: M | Risk: Low-Medium | Completeness: 7/10

### Approach C: Hybrid — Shared Core + Two Frontends
Extract atomic pre/post-processing catalog into a shared core. model2webnn consumes this core for WebNN. webai.js is the unified CLI/Web UI that adds ORT Web and LiteRT.js inference emitters.
Effort: M-L | Risk: Low | Completeness: 9/10

### Preprocessing Resolution Order

When a user runs `webai generate --task image-classification --model ./my-model.onnx --runtime ort`:

1. **Read model input shape** — For local ONNX/TFLite files, parse the model's input tensor metadata (shape, dtype) directly from the file. ONNX and TFLite both encode this. Use it to infer image dimensions and channel layout.
2. **Check for HF config** — If model is a HuggingFace ID, fetch `preprocessor_config.json`. If found, parse exact parameters (image_size, image_mean, image_std, do_resize, do_normalize, etc.)
3. **Check model-profiles** — If model name matches a known profile (e.g., "mediapipe-hand-detection", "yolov8", "vit-base"), use those specific parameters.
4. **Fall back to task defaults** — Use sensible defaults for the task type (e.g., image-classification defaults: resize to 224x224, normalize with ImageNet mean/std, NCHW layout). Emit a CLI warning: "Using task defaults — verify preprocessing matches your model."
5. **CLI overrides** — `--image-size 640 --normalize-mean 0,0,0 --normalize-std 1,1,1` flags override any resolved values.

The resolution cascade is implemented in `src/config/preprocess-resolver.ts`, which takes (model path/ID, task type, CLI flags) and returns a fully-resolved `PreprocessConfig` object. Each step can fill or override fields; later steps take precedence.

**Compact mode** (`--mode compact`): Functions inlined into a single async function, comments removed, but variable names preserved for debuggability. Not run through a minifier — the user's bundler handles that. Production-ready drop-in.

**TypeScript mode** (`--lang ts`): Same structure as JS output, with type annotations on all function parameters and return types, typed tensor constructors, and a `tsconfig.json` snippet showing required compiler options (target: ES2020, lib: DOM). Default output is JavaScript; TS is opt-in.

### Tree-Shaking by Construction

The user asked: "When user include this project in their web application source code, anyway to not include not used codes?"

The answer is built into the architecture:

1. **Code generation mode (primary):** Only the functions for your specific task are emitted. There's nothing to tree-shake because unused code was never generated.
2. **Library import mode (optional):** If `@webai/core` is published as an npm package, each preprocessing function is a separate ESM export. `import { resizeImage, normalizeToNCHW } from '@webai/core/preprocess'` — bundlers tree-shake the rest.
3. **Compact mode:** Inlines only used functions into a single self-contained function. Zero dead code by definition.

## Error Handling Strategy

**CLI-time errors** (webai.js handles these):
- Invalid model path/ID: fail fast with clear message ("Model not found: ...")
- Network failures during HF config fetch: warn and fall back to task defaults
- Unsupported task type: list supported tasks
- Incompatible runtime + model format (e.g., WebNN with a model that has unsupported ops): warn with specifics

**Generated code runtime errors** (the generated code's responsibility):
- Generated code does NOT include try/catch — that's the user's application concern. The generated code is a building block, not a framework.
- Exception: `--runtime webnn` generates a capability check (`if (!navigator.ml)`) with a console warning suggesting fallback to ORT Web. This is the one case where the tool generates error handling because WebNN availability is uncertain.

## Tokenizer Strategy (Phase 2 gate)

Full BPE/WordPiece tokenization requires vocabulary files (potentially megabytes) and merge rules. Three options, to be decided before Phase 2:

**Option A (recommended): Generate code that loads `tokenizer.json` at runtime from HuggingFace.** The generated JS includes a lightweight BPE decoder (~200 lines) that consumes the standard HuggingFace `tokenizer.json` format. The vocab/merges stay as a fetched JSON file, not embedded in code. This matches how Transformers.js handles it — but without the class hierarchy.

**Option B:** For simple models (sentiment analysis, etc.), generate a basic whitespace + vocabulary lookup tokenizer that embeds a small vocab as a JS object. Works for models that don't need subword tokenization.

**Option C:** Emit a wrapper that delegates to `@huggingface/tokenizers-wasm` (the Rust tokenizer compiled to WASM). Small binary, correct behavior, but adds a WASM dependency.

## Audio Preprocessing Risk

Audio tasks (ASR, audio classification) require Mel spectrogram / MFCC computation, which depends on FFT. This is the highest-risk item in the preprocessing catalog:

- FFT in pure JS on typed arrays is ~150 lines and correct but slow for long audio
- Web Audio API's `AnalyserNode` provides FFT but requires an AudioContext and doesn't work in Web Workers
- A vendored radix-2 FFT implementation is the pragmatic choice — self-contained, no dependencies, adequate performance for inference preprocessing

Audio tasks are intentionally placed in Phase 2.5 to allow this complexity to be addressed after the core architecture is proven with CV tasks.

## WebNN-Specific Optimizations

When `--runtime webnn` is selected, the generated code should include performance optimizations from Microsoft's webnn-developer-preview (reference: `/references/webnn-developer-preview`). These patterns are critical for real-world WebNN performance:

1. **Pre-allocated MLTensor reuse.** Create MLTensor objects once during initialization and reuse them across inference calls. Avoid creating/destroying tensors per inference. For LLMs, pre-allocate the entire KV cache upfront:
   ```javascript
   // Allocate once
   const inputTensor = await mlContext.createTensor({ dataType: 'float32', shape: [1, 3, 224, 224], writable: true });
   const outputTensor = await mlContext.createTensor({ dataType: 'float32', shape: [1, 1000], readable: true });
   // Reuse across inferences
   mlContext.writeTensor(inputTensor, preprocessedData);
   mlContext.dispatch(mlGraph, { input: inputTensor }, { output: outputTensor });
   ```

2. **Tensor swap pattern (LLMs).** After each decode step, swap feed/fetch tensors instead of deallocating:
   ```javascript
   feed[`past_key.${i}`] = fetches[`present_key.${i}`];  // Reuse, don't recreate
   ```

3. **Fixed shape strategy with padding (current).** WebNN currently doesn't support dynamic input shapes. Fix shapes at model load time via `freeDimensionOverrides`, pad variable-length inputs to maxLength. Avoids expensive graph recompilation. **Note:** Dynamic input shape support is under active W3C discussion (webmachinelearning/webnn#883) with a Chromium implementation POC in progress. Once shipped, the code generator should detect browser support and emit dynamic-shape code when available, falling back to fixed-shape + padding for older browsers.

4. **Dual session pattern (LLMs).** Create separate prefill (variable sequence length, padded) and decode (sequence_length=1) sessions. Both created once, reused for all inferences.

5. **Pre-allocated host-side read-back buffers.** Pre-allocate typed arrays (Float32Array / Float16Array) for reading results back from MLTensor, avoiding allocation per read.

6. **Explicit tensor lifecycle management.** Call `mlTensor.destroy()` when done to free GPU/device memory. Call `session.release()` and `mlContext.destroy()` at cleanup.

7. **MLContext reuse.** Create the MLContext once at initialization, reuse across all tensor creation and inference operations.

The code generator should emit these patterns when `--runtime webnn` is selected. For simple single-inference cases (image classification), include patterns 1, 5, 6, 7. For streaming/iterative inference (LLMs, video), include all patterns.

**Reference:** `webnn-developer-preview` is also added as a reference project in the repo.

## Resolved Decisions

1. **Tokenizer strategy:** Option A — generate code that loads `tokenizer.json` at runtime from HuggingFace, with a lightweight BPE decoder (~200 lines). See Tokenizer Strategy section above.
2. **model2webnn integration:** webai.js imports model2webnn's `convert()` as a library (npm dependency) for `--runtime webnn`. See Architecture section.
3. **Audio preprocessing:** Vendored radix-2 FFT implementation for Mel/MFCC. See Audio Preprocessing Risk section above.

## Open Questions

1. **Model weight hosting:** For the "HuggingFace model ID" flow, the generated code needs to point to hosted model weights. Should it reference HuggingFace CDN URLs directly, or should the user provide their own hosting?
2. **WebNN availability:** WebNN is still Chrome-only and flag-gated. The generated WebNN code includes a capability check. How graceful should the fallback be — console warning, or auto-switch to ORT Web?
